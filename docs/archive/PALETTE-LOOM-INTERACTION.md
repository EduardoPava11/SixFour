# The Palette Loom — Authoritative Interaction Design

**The 16×16 hand-merge palette-authoring experience for SixFour.** The user *builds*
the global 256-color palette by hand, folding it toward the look they want. `B(A(L))`
is **scaffold, not automator.** Status of record: 2026-05-30.

> Produced by the `lnn-merge-authoring-interaction` workflow (6 mechanics decomposed →
> adversarially verified → 2 journeys incl. an anti-automation stress test → synthesis).
> All 6 mechanics survived (REVISE, none CUT). Supersedes the *passive* UI of
> `L-NN-PRODUCT-ABSTRACTION.md` (its construct/traceability mapping still holds); the
> `B(A(L))` cascade in `L-NN-ATOM-DESIGN.md` is unchanged.

> **Phase gate up front (this resolves the verifier consensus).** The deployed head is
> the **depth-8 grayscale L head** (`HANDOFF §1/§63`): every shipped cell is
> `(L_i, 0, 0)`, a=b=0. For a grey color `σ(c)=(L,−0,−0)=c` — **a grey is its own
> complement**, so in Phase L there is *nothing to pair* and "merge toward grey" is
> vacuous (every cell is already grey). Therefore:
> - **Phase L (today, shippable) — the Tonal Loom:** hand-author a **tonal
>   posterization** by fusing lightness levels up an **L-bisection / Lloyd-Max merge
>   tree**. 256 → a user-chosen coarser tonal set. *No pairs, no complements, no false
>   pair-promise.*
> - **Phase A/B (future, NOT-YET-BUILT) — the Sigma-Pair Loom:** once chroma deploys,
>   `a,b ≠ 0` ⇒ `σ(c) ≠ c` and **true complements exist**. *Only then* do the 128
>   σ-pairs, merge-to-grey, and the 256-leaf→128-pair morph become real. Labeled future
>   throughout; **palette-swap-away**.
>
> This binds **256 / depth-8 to L (today)** and **128 / depth-7 to the σ-pairs (chroma,
> future)** instead of conflating them — fixing the 128-vs-256 / depth-7-vs-depth-8
> conflict (`PRODUCT §6`, `ATOM §2/§6.1`).

## 1. The thesis: the user builds the palette

The global 256-color palette is not handed down by an argmin — it is **hand-authored by
merging**, and the labor IS the product. The user is given the raw material (256 cells
laid as a 16×16 loom) and physically **folds** it: fusing tonal levels by taste in
Phase L, and (when chroma ships) authoring complementary harmonies in Phase A/B. The
network `B(A(L))` **scaffolds** — it grounds tone, guarantees every fold lands on a
valid tonally-anchored color, and may *suggest* — but it **never performs the
collapse.** Research is the reason, not a constraint we resent: the deterministic
Lloyd-Max collapse already owns fidelity (the NN *cannot beat it* on MSE), so the net's
only honest value was always **empowering the human author**. There is no
"auto-collapse" button, by deliberate design. **The user MUST do the work**, and two
users get genuinely different looks from the same burst because the folds are theirs.

---

## 2. The merge mechanic

**The core verb is MERGE — `2→1` Haar synthesis, one fold at a time.**

**What a merge does (the math).** Merging two cells `c₀, c₁` produces their **Haar
midpoint** parent and retains the half-difference detail:

```
parent  m = (c₀ + c₁) / 2        -- Haar low-pass (the new color shown on screen)
detail  d = (c₀ − c₁) / 2        -- retained, never discarded (lossless synthesis)
split(m,d) → { m+d , m−d } = { c₀ , c₁ }   -- exact inverse (Haar analysis)
```

- **Phase L (today):** cells are `(L,0,0)`. Merge ⇒ `m = ((L₀+L₁)/2, 0, 0)` (a wider
  tonal band) and `d = ((L₀−L₁)/2, 0, 0)` — the detail's nonzero component is in **L
  (σ-symmetric)** and is **honestly labeled a lightness deviation, NOT chroma.** This
  climbs an **L-bisection tree** toward a coarser tonal set. There is **no grey
  target** — every cell is already grey, so a grey magnet would be vacuous.
- **Phase A/B (future):** when `a,b ≠ 0`, the special case **complement-merge**
  `c ⊕ σ(c)` gives `m = (L,0,0)` (pure grey anchor) and `d = (0,a,b)` (pure chroma) —
  `c + σ(c) = 2·(L,0,0)`. Grey becomes a *consequence the user feels*, and (per §6)
  σ-pairing is generated by a **fixed operator**, not by hunting an absent mirror.

**Gesture.** Press-and-hold a cell — it lifts, casts a shadow, the other 255 dim — then
**drag it onto a second cell and release**. Single-tap is **inert by design** (the fold
must be unambiguous on a small grid). Drag-with-confirm or two-finger pinch; never a
marquee that batches decisions away.

**Feedback.** The fused band re-tiles in place; a counter ticks (`256 tonal levels ·
your set: 255`, honest — *not* `0/128 pairs` in Phase L). The retained `|d|` shows as a
small bar (how far apart the fused levels were).

**Reversibility (by hand, not by an undo stack).** **Long-press → pull apart**
(two-finger spread) splits a parent back to its exact two children via `split(m,d)` —
lossless, because `d` was retained at merge time. Re-authoring is hand-craft, not an
"undo" that erases the labor.

**`2→1` vs `n→1`.** The default and only primitive is `2→1`. A two-finger gather / lasso
is **syntactic sugar = a deterministic left-fold of pairwise `2→1` midpoints in
selection order**, each step animated, individually undoable, replayable. `n→1` is never
a black-box reducer and never a new primitive — it keeps the Haar math and reversibility
intact, reducing *scanning* effort, never *decision count*.

## 3. The 256-morphology

The grid is one cardinality — `256 = 16² = 2⁸ = 4⁴ = 256 SIMD lanes = the 64³ voxel
global-palette target` — read through different factorization **faces**. As the user
merges, the loom climbs from the flat face toward the terminal shape. **The shape is a
live readout of how much hand-work remains.**

| Face | Reading | Worker meaning |
|---|---|---|
| **16² flat** | 256 loose cells, dark→light, raw material | "the mat — read it before you fold" |
| **4⁴ quadtree** | 2×2 super-cells coarsen in powers of 4 (256→64→16→4→1); cell *area* grows with fold depth | "posterization depth / tonal richness — chunkier = flatter" |
| **2⁸ binary spine** | balanced Haar cascade, parents nested above their two children | "ordered pairwise folds — one merge step at a time" |
| **terminal** | **Phase L:** a list of N user-chosen **tonal bands** (no forced count). **Phase A/B:** a **128-long list of σ-pairs** (depth-7), each row = anchor `mᵢ` flanked by its mirror | "the look you authored" |

**Direction matters (verifier fix).** `256 → 128` is the **first rung up** the binary
tree, not the terminal. The quadtree (64 nodes) and deeper spine are **coarser** than
128. In Phase L there is **no forced 128 target** — the user folds to the richness they
want. In Phase A/B the destination 128 is reached by **one binary level** (`2⁸→2⁷`); the
4⁴ quadtree is a *navigation view grouping* of the result, not the pairing arithmetic
(`128 ∉ {4ᵏ}`).

```
PHASE L (today) — the Tonal Loom (no pairs):

  16×16 FLAT (256 L-levels)          climb the L-bisection tree
  ┌────────────────┐                        by taste
  │░░▒▒▒▓▓▓▓████████│   merge      ┌──────────────┐   merge     ┌────────┐
  │░░▒▒▒▓▓▓▓████████│  ───────▶    │ ░░ ▒▒ ▓▓ ██  │  ───────▶   │░▒▓█│   N
  │  …16 rows…      │  (fuse       │  broader      │  (fuse      │ user-  │   tonal
  │░░▒▒▒▓▓▓▓████████│   adjacent   │  bands)       │   more)     │ chosen │   bands
  └────────────────┘   levels)    └──────────────┘             └────────┘
   256 = 16²              ↑ quadtree (4⁴) / binary (2⁸) views are lenses on the climb
        no grey dot · counter: "256 tonal levels → your set" · NO 0/128 promise


PHASE A/B (future, NOT-YET-BUILT) — the Sigma-Pair Loom:

  256 leaves (L,a,b)        one binary rung up (2⁸ → 2⁷)        128 σ-PAIRS
  ┌────────────────┐                                          ┌──────────┐
  │ c · σc generated│   author spread dᵢ on a/b disc;          │ m₀  ◀▶   │  each row:
  │  by σ (free)    │   σ mirrors it equal-&-opposite;  ───▶   │ m₁  ◀▶   │  anchor mᵢ
  │  user steers d  │   grey = the felt consequence            │  …128…   │  + exact mirror
  └────────────────┘                                          └──────────┘
        σ-pair structure drawn for free · USER authors d, not the pairing
```

## 5. The user journey

She opens **Review**. The 64-frame GIF loops; below it the **Palette Loom** fills —
**16×16, 256 cells**, grayscale leaves from the deployed L head, row-major dark→light.
The counter is honest: *"256 tonal levels · your set: 256."* No `0/128` promise — chroma
hasn't shipped, so there are no complements to fake. She **reads the mat** first: where
darks crowd, where highlights need air. The grid is **inert** until her finger moves.

A second user, Dev, opens the same screen impatient — thumb straight to the corners
hunting **"Auto-pair all."** There isn't one. He taps a swatch: nothing. **Single-tap is
inert by design** — *merge is press-hold-drag.* The design forces the work before he can
skip it.

She **picks up** a near-black cell (press-and-hold — it lifts, the other 255 dim) and
**drags it onto its neighbor** (the **merge** verb). Haar synthesis fires:
`m=(L₀+L₁)/2`, detail `dL=(L₀−L₁)/2` retained, lossless. Two cells fuse into one wider
band; counter ticks to **255**. She feels the fold land.

She climbs *by taste* (the **tonal-grouping** mechanic) — fusing adjacent darks she
wants flat, leaving highlights finely separated. No quantizer makes this call. As a
**4×4 quadrant resolves**, the block snaps from the **16² flat face** toward the **2⁸
binary spine** — she *triggers* the reshape by finishing the work, not by tapping
"next."

Dev tries to cheat the tedium with a **two-finger lasso**. It doesn't black-box-reduce;
it replays as a visible **left-fold of `2→1` midpoints**, each step animated and
undoable. Batched, still his hand — no shortcut, but no tedium.

The complexity slider whispers *"the scene asked for ~40 levels"* — a **dismissible
hint** (the E[d] readout). She ignores it; it never auto-folds. A fingertip slip —
**long-press, pull apart** — `split(m,d)` restores both children exactly. Undone by
hand, not by an undo stack that erases her labor.

She settles at the richness *she* wanted: the 16×16 morphed into broad, deliberate
bands, dark→light. Dev, having found no button, has authored his own coarser set —
**different from hers, from the same burst.** Both tap **export**. The GIF re-renders
against each authored posterization — 64³ voxels carrying *their* tonal skeleton. The
work mattered because the look is theirs: every band a decision no machine would have
made the same way.

## 6. NN as scaffold, not automator

**The exact line.** `B(A(L))` is forbidden from collapsing the palette. It does three
things and suggests one — it **never actuates a merge.**

**What it GUARANTEES (scaffold):**
1. **Tone grounding.** L runs the `64→1` permutation-invariant pool and the
   **deterministic argmin / Lloyd-Max** so the 256 leaves the user is handed are
   fidelity-anchored and every cell is a **populated, significant** L-level
   (`SignificantVoxelVolume`). The row-major L-sort is L's ordering. *Phase L: `d=0` for
   every leaf — they sit on the grey skeleton.*
2. **Valid-complement legality (Phase A/B).** Because A,B write **only** the
   σ-antisymmetric eigenspace and L **only** the symmetric one (`symPart(look)=L`,
   `asymPart(L)=0`, orthogonality), every authored pair `{mᵢ±dᵢ}` is **provably
   complementary by construction** — the user cannot author an off-manifold harmony.
   Crucially this **does not require the input to be σ-closed**: the mirror is
   *generated* by the fixed σ operator, **never hunted** in a non-symmetric pooled grid
   (this was the fatal break the verifiers flagged — σ-pairs live in the *output*, not
   the input).
3. **In-gamut anchors.** Every user midpoint lands on a real tonal anchor; L tells the
   UI where neutral truly sits.

**What it may SUGGEST (dismissible, inert until acted on):**
- Phase L: the **E[d] complexity hint** ("~K levels") — a non-committal whisper, never
  an auto-fold.
- Phase A/B: on an **explicit long-press only**, a **de-fanged oracle** showing the
  **top-3 candidate mirrors with their residual ε** (real captures rarely contain an
  exact `σ(c)`, so an inexact harmony lands at `(L, ε_a, ε_b)` — surfaced as a visible
  off-grey tint so the math never lies). The user still **judges which mirror** is the
  harmony they want. *Never auto-surfaced, never a one-tap-per-pair clicker walk.*

**What the USER ALONE authors:** which levels to fuse and how deep (the tonal
posterization, Phase L); the chroma spread `dᵢ` of each harmony (Phase A/B) — the
relational, taste-dependent freedom with **no closed-form optimum.** σ may *reflect* the
user's `d` (a fixed operator), but the net **never computes a midpoint, never authors a
`d`, never fills an absent mirror.**

**NO AUTO-COLLAPSE BUTTON.** There is no "pair all," "build my palette," or "let the NN
finish." The anti-automation principle is **enforced by absence** and by the gesture
grammar: the *only* path to the authored palette is performing the folds by hand. The
merge actuator is wired to the finger; the NN is structurally a scaffold layer with no
merge actuator.

## 7. Engineering implications

- **Merge = user-driven Haar synthesis.** The fold op is exactly `analyze`/`pairReduce`'s
  `m=0.5*(c₀+c₁)`, `d=0.5*(c₀−c₁)` (= `SigmaDecomp.symPart`/`asymPart`; = `AxisNet.greyOf`
  on the L range), with `split` as its exact inverse. **Implement merge/split as a thin
  Swift wrapper over the existing Haar kernels** — the math already ships verified; the
  new code is *gesture → which two cells → call synthesis*, plus a retained-`d` store for
  lossless reversibility. No new numerics, no new golden vectors for Phase L.

- **The 256-grid IS the deployed global palette, not a per-frame palette.** Render the
  loom from the L head's **256 grayscale OKLab leaves → `s4_palette_oklab_to_srgb8`** (the
  same path `HANDOFF §④` uses for assembly). The grid maps **1:1 to the shippable global
  palette / 256 SIMD lanes / 64³ voxel target** — folding it *is* editing the global LCT.
  (The 64 per-frame 256-palettes remain the NN *input*, per `CLAUDE.md`; do not confuse
  the two surfaces.)

- **The σ-pair is the *Phase-A/B* merge target, generated not hunted.** When chroma
  lands, expose `reconstructPaired`'s `mᵢ` and `dᵢ` **separately** so the UI renders the
  anchor + steers the spread, with σ mirroring for free. Do **not** present the decoder's
  internal `[c₀,σc₀,…]` adjacency as a "find the partner" task — that pre-solves the work.
  The trainer must emit `mᵢ`, `dᵢ` as distinct tensors (the `SigmaPairHead` 384-DOF tree,
  factored per `ATOM §1.1`).

- **What the trainer/spec must expose so the USER performs the collapse.** (1) The **256
  L-leaves** (Phase L) and, later, the **128 anchors `mᵢ` + spreads `dᵢ`** (Phase A/B) as
  addressable, re-renderable arrays. (2) A **forward-golden for the L head** (extend
  `Codegen.Golden`/`axisnet_golden.json`) so the device-rendered loom is bit-checked. (3)
  The **global re-index stage** (`global_reindex`, `HANDOFF §③`) must run against the
  **user's authored palette**, not the net's raw output — re-index is a *post-authoring*
  render stage. The net stops at *grounding the leaves*; the index map and L-anchors stay
  **non-editable** (the single-right-answer parts are walled off so the user's whole
  effort lands on the no-single-answer choices).

- **Anti-automation is a build constraint, not just UX.** There must be **no kernel/API
  that maps 256 → final palette in one call** exposed to the UI. `n→1` lasso compiles to a
  replayable sequence of `2→1` synthesis calls (order recorded, each reversible). Testable:
  assert the authored palette is reachable *only* via a recorded fold sequence.

- **`L-NN-ATOM-DESIGN.md §5` staged refactor — mostly unchanged.** The E→R→D atom
  factoring (separating `L`, `A`, `B` into the three eigenspaces) is *exactly* what this
  interaction needs: it gives the UI separately-addressable `m` (L atom) and `d` (A/B
  atoms). **Keep the staged refactor as-is**; this design adds a *consumer* of its outputs
  (the loom), not a change to the cascade. The one addition: the refactor's deliverable
  list should include "emit `mᵢ` and `dᵢ` as distinct, UI-addressable tensors."

- **The depth-8/256 vs depth-7/128 conflict is resolved at the head by phase.** Today's
  **deployed depth-8 L head = 256 distinct L-levels** is the correct **Phase-L** merge
  source (256 tonal levels → user-chosen coarser set). The **depth-7 / 128-anchor σ-pair
  head** (`ATOM §2/§6.1`) is the correct **Phase-A/B** merge target (128 complementary
  harmonies). They are **orthogonal axes** (256 = L-resolution count; 128 = pair count) —
  bind each to its phase rather than conflating them. **Action before the Phase-A/B loom:**
  wire the `SigmaPairHead` 384-DOF pivot and reconcile head selection. **Phase L ships now
  against depth-8/256; no head change required for the Tonal Loom.**
