# What the L-NN Is FOR — and How Its Three Forms of 256 Serve the User

> **Status of record: 2026-05-30.** This is the authoritative *function-design* document. It defines the look-NN by what it DOES FOR THE USER and lets the three forms of 256 (16² / 4⁴ / 2⁸) follow. It is orthogonal to the decoder-shape decision. Ground truth: `spec/BLEED_LOOP.md` (obfuscate/reveal, Def 45–57, Thm 14–18), `spec/LOOK_NN.md` (collapse, Def 30/Thm 9), `spec/src/SixFour/Spec/Obfuscation.hs` (landed keystone), `spec/src/SixFour/Spec/{PairTree,SigmaPairHead,Collapse,Bures,Preference,Significance}.hs`. Where a function depends on un-landed math it is marked **[SPEC]** (contract-first, dormant) versus **[LANDED]**. Companion to `docs/L-NN-MASTER-DESIGN.md` (the design of record); this doc is the *function* layer beneath it.

---

## 1. What the NN is FOR (the function thesis)

**The look-NN is a One-Look Maker.** The job-to-be-done, in one sentence: *take the user's 64-frame colour burst and hand back ONE coherent global 256-colour look that the user AUTHORS and OWNS — never 64 flickering per-frame grades, and never a correctness problem the user has to babysit.* The user gets a GIF and its 256-colour palette that reads as a single deliberate grade across the whole burst, plus an authoring surface where every move they make is about the LOOK, never the fidelity. Three capabilities deliver this, in order. **GROUND** [LANDED]: the NN pools the 64 per-frame palettes into ONE global palette (permutation-invariant Wasserstein/Bures barycenter, `LOOK_NN` Def 30/Thm 9 = population-weighted Lloyd k-means on pooled candidates, `Collapse.hs`/`Bures.hs`) and places every anchor at the Lloyd–Max ceiling behind a deterministic argmin index map (`LookNet.nearestIdx`/`Significance.nearestCentroid`, every cell ≥ `minPopulation` pixels). Fidelity is then *walled off* — there is one right answer the user cannot get wrong — so the user's whole effort lands on the no-single-answer LOOK. **OBFUSCATE/REVEAL** [keystone LANDED, reveal SPEC]: every cell ships as grey `(L,0,0)` but its chroma `(0,a,b)` is *banked, not deleted* (`Obfuscation.hs`, Thm 14 lossless Ω, law `L45.1` grayscale-truth, 332 tests green); the user authors TONE first on the grey skeleton, and a later bleed dial βₜ progressively REVEALS the retained chroma (Thm 15, `ColorBleed`/`ChromaAllocation` are spec'd but not yet built — dormant per `BLEED_LOOP` §6 Gap 7). **SCAFFOLD-NOT-AUTOMATE** [LANDED discipline]: the NN suggests and never commits — `S_θ` is a *section* (`π∘S_θ = id` on committed coordinates, Def 57), the user's fold `F_U` is the only writer (Thm 16), and there is deliberately *no auto-collapse button* (anti-automation by absence). The loop's attractor is **creation, not an MSE optimum** (Thm 17): each iterate invites the next.

## 2. The three forms, three functions (the form-follows-function map)

The three forms of 256 (256 = 16² = 4⁴ = 2⁸) are GIVEN and useful. Each is the hand-grip for exactly one user function.

| FORM | user FUNCTION | what the user DOES | what the NN does to HELP | why the form affords the function |
|---|---|---|---|---|
| **16²** — the 16×16 loom grid Γ (`BLEED_LOOP` §2) | **SEEING** | Scans all 256 cells at once as one image; locates a tone by *where it sits in the dark→light sweep*; compares two cells by glancing between positions; taps a cell to inspect / select-for-fold / hand to the drill | COLLAPSE: pools 64 per-frame palettes into the ONE global palette the grid shows. GROUND: deterministic argmin + Lloyd–Max + significance make every cell a real, populated, tonally-anchored representative — the user reads truth, not a smear | A 16×16 grid shows all 256 in one foveal sweep with a home cell per slot (16²=256=K, no gaps/overflow). A list has no overview; a 1D strip wastes the second axis. Position carries the **L-ordering** so neighbours-on-grid are neighbours-in-tone — "scan as an image" is literally true for the tonal skeleton |
| **4⁴** — the quadtree (256→64→16→4→1) | **NAVIGATING** | Pulls the palette coarse-to-fine to choose richness vs posterization; taps a 2×2 super-cell open to descend into one region (the crowded darks, the highlights) and hand-folds *locally* there | GROUND every level so each coarsened super-cell is a real populated anchor. PROPOSE (dismissibly) which region the scene's histogram is asking for work in. The user's *committed* folds compile to the lossless **binary** spine, never lost | Powers-of-4 over a 16×16 mat reads as physical BLOCK SIZE — chunkier super-cell = more fused = flatter look. Coarse-to-fine drill IS a quadtree; a flat grid has one fixed pitch, a binary spine reads less cleanly as five legible granularity stops |
| **2⁸** — the binary / Haar PairTree (8 levels) | **AUTHORING** | FOLDS pairs (press-hold-drag one cell onto another → they fuse to the parent) and SPLITS them back apart (long-press → pull apart) until the look is posterized to THEIR chosen richness, then emits the GIF | GUARANTEE the fold is lossless: m=½(c₀+c₁) banking d=½(c₀−c₁) is exactly `PairTree.analyze`, split is `reconstruct` (round-trip exact). RETAIN the banked d for a later reveal. SUGGEST the next fold via E[d] / `goldenDecay` — dismissibly | AUTHORING is making decisions, and the smallest commit is *binary* — this pair → one. 2⁸ is the only form whose primitive is a pair, so the only one that natively affords a single-gesture commit. The Haar average/detail (m,d) IS the lossless fold/split, so undo is free |

The forms are not decoration: each is the surface one capability of the single job needs (see the floor → navigate granularity → author the fold).

## 3. Seeing — the 16×16 grid (the Loom-Map)

**The function:** one screen that shows the user the *entire 256-colour global palette at once* as a browsable 16×16 surface — the loom Γ. This is the canonical "where is my look" home of the loop. The user reads the whole look in one glance, locates a tone by position, compares any two cells by glancing between them, and taps any cell to pick / inspect / select-for-fold (handing off to the 4⁴ drill or staging a 2⁸ fold).

**The meaningful layout the NN grounds — and what it is NOT.** The grid is sorted by **lightness, row-major dark→light**, exactly as the deployed L head emits it (`PALETTE-LOOM §5`; the Phase-L loom is a *lightness*-Haar tree, `BLEED_LOOP` §3.4). **Position means TONE:** x and y carry the L-ordering of the 256 cells and each cell renders at its own lightness, so neighbours on the grid are neighbours in tone and "scan as an image" is literally true for the tonal skeleton. The grid is **not** an a–b chroma embedding and **not** a trained embedding — the joint a–b chroma authoring lives on a *separate* chroma disc (Phase A/B, settled background), which must never be folded into this grid. (This corrects an earlier draft that mislabelled the loom axes as AxisA/AxisB: in the only shippable state every cell is `(L,0,0)`, a=b=0, so there is no a–b plane to lay out here — `Obfuscation.hs` L45.6 grey-vacuity.)

**The 2D chroma surface, honestly placed.** The "see all the chroma at once" affordance is real but it is the *reveal*, not the layout. At β=0 (the shipped state) the surface is the honest grayscale tonal map. As the user drives the bleed slider, the chroma that the NN losslessly BANKED behind each cell (R c, Ω-retained) reveals **in place** — same cell, same position, colour filling in — because the grey skeleton, and thus every cell's location, is β-invariant (`Obfuscation` L48.2, Thm 16). So the user reads pure TONE order first, then watches retained colour appear without the map moving. **[SPEC]** β>0 is dormant until the σ-pair head is wired (`BLEED_LOOP` §6 Gap 7).

**What the NN does here:** (1) COLLAPSE — without it there are 64 flickering grids; with it there is one map to read. (2) GROUND — the argmin index map (Thm 18) and Lloyd–Max placement make each cell the single correct representative, so the user never fixes correctness, only reads/authors the look. (3) it may dismissibly suggest a region worth attention but **never re-sorts, auto-picks, or auto-folds** (L52.10, anti-automation by absence).

## 4. Navigating — the 4⁴ quadtree (drill-down / richness / zoom)

**The function:** NAVIGATE the look's granularity by drilling the palette coarse-to-fine. The verb is the existing loom MERGE (press-hold-drag); the 4-up quad view is the **lens** that lets the user aim that fold at a region of colour-space instead of scanning 256 loose cells. The user does two concrete things: (1) **ZOOM the richness** — pull the whole palette up or down the quad-ladder by taste: coarser = chunkier / more posterized / fewer effective tones (flatter, graphic); finer = richer, more separated (256 at the floor), surfaced as cell AREA. (2) **DRILL a quadrant** — tap a 2×2 super-cell open (256→64→16→4) to descend into one region and hand-fold *locally*: fuse the crowded darks flat while leaving a different quadrant's highlights finely separated. Single-tap zooms; the fold itself stays a deliberate drag. There is no auto-posterize button.

**What the NN does:** (1) GROUNDS every level so each coarsened super-cell is a real, populated, tonally-anchored colour (argmin + Lloyd–Max + `SignificantVoxelVolume`, every cell ≥ `minPopulation`) — zooming to a coarse level reads TRUTH, not an averaged smear. (2) PROPOSES the drill target dismissibly: the E[d] complexity readout whispers "the scene asked for ~K levels," and the retained-chroma energy χ_p flags where un-revealed colour is concentrated — a hint of *where* drilling pays off, **never a fold**. (3) GUARANTEES the folds the user *commits* are lossless and reversible.

**Honest grounding (the substrate vs. the lens).** Lossless reversibility and the m=½(c₀+c₁) / retain-d merge live on the **2⁸ binary spine** (`PairTree`; `BLEED_LOOP` Def 50, Thm 14) — that is what the user's committed folds compile to. The 4⁴ quad view is a chosen 4-ary **presentation** over those folds. Per the landed code (`Quad4.hs`, `Quad4Fit.hs`), Quad4 is a structurally *distinct* 513-DOF alternative — it injects into `HaarPalette` (768→513) and can be **lossy** (`quad4Residual`; rank(B) may be < 384, so "Quad4 cannot even fit all σ-pair palettes"), and Quad4-vs-PairTree is an *unresolved* measurement question. So this function promises **lossless BINARY folds shown through a 4-up navigation lens** — not a "free lossless quad regroup." Dropped as ungrounded: the "single posterization dial maps one-to-one onto quad depth" claim and treating χ_p as a quad-drill priority metric (χ_p is defined over the 128 σ-pairs and belongs to the orthogonal bleed/reveal axis, `BLEED_LOOP` Def 51 — borrowed here only as a dismissible "work remains here" hint).

**Why the form:** navigation is inherently spatial coarse-to-fine, and powers-of-4 is the geometry of a 2-D surface — each level is a 2×2 super-cell over the 16×16 mat, so the eye reads granularity as block size. (It is the SAME 256 colours regrouped, not a render-resolution change — the genuine 64³ render funnel is a separate orthogonal dial.)

## 5. Authoring — the 2⁸ binary / Haar tree (the Fold)

**The function — THE FOLD.** The user AUTHORS the global 256-colour look by hand-folding the 16×16 grid one binary 2→1 merge at a time, until the palette is posterized to the richness THEY chose, then emits the GIF. In the shippable Phase-L the user does two binary acts:

- **FOLD** — press-hold-drag one cell onto another; they fuse into the Haar parent **m = ½(c₀+c₁)** shown on screen, and the half-difference detail **d = ½(c₀−c₁)** is *banked* (the tonal band widens by one notch; the counter ticks 256→…→N).
- **SPLIT** — long-press → pull a fused cell apart, re-dividing it into its exact two children m+d, m−d, *losslessly* re-opening that decision.

There is **no forced fold-count and no fixed depth to climb**: the user folds adjacent darks they want flat and leaves highlights finely separated, settling at THEIR posterization — two users get different looks from the same burst (`PALETTE-LOOM §3`: "in Phase L there is NO forced 128 target"). The single binary fold gesture is the whole authoring primitive — no typing coordinates, no per-colour sliders, no auto-collapse button.

**What the NN does (scaffold, never actuator):** (1) GROUNDS the leaves — the L-atom pools the 64 per-frame palettes (permutation-invariant 64→1) and the deterministic argmin/Lloyd–Max places every cell as a real, significant tonal anchor, freeing the user to author only the look. (2) Makes fold/split LOSSLESS — m=½(c₀+c₁) retaining d=½(c₀−c₁) is exactly `PairTree.analyze`, split is `reconstruct` (`lawReconstructAnalyzeRoundTrip`); d is banked, never deleted (Obfuscation Thm 14, fold = Ω at t=0), so undo is free *and* chroma is retained for a later reveal. (3) SUGGESTS dismissibly — the E[d] complexity readout whispers "~K levels" and `goldenDecay` orders coarse-to-fine, but it **never auto-folds** (L52.10 returns only an admissibility predicate + a dismissible suggestion). (4) Anti-automation is a BUILD constraint — there is no 256→final-palette kernel exposed to the UI, so the only path to the authored palette is the user's own recorded fold sequence.

**Why the binary (2⁸) form.** The 2⁸ face is the **gesture grammar** — the math of the fold verb itself (one 2→1 Haar synthesis, exact split as inverse), the smallest possible commit (this pair → one). AUTHORING is making decisions and the smallest decision is binary; 2⁸ is the only one of the three forms whose primitive is a pair, so the only one that natively affords a single-gesture commit (a 4-way quad node is too coarse for a clean this/that). **It is NOT a fixed 8-level harmony tree the user climbs** — the 8-level depth is the math of the fold verb, *shared by every step*, not a ladder the user walks down (`L-NN-MASTER-DESIGN`/`BLEED_LOOP` §3.4). `goldenDecay`, the 384-DOF design matrix, and the 1/φ detail schedule are **NN-internal** and stay off the user surface.

**[SPEC] — FUTURE, Phase A/B (gated, not shipped).** In the shippable Phase-L every cell is grey, σ(c)=c, "a grey is its own complement" (`Obfuscation` L45.6) — so there are **no σ-pairs and nothing to pick**. Once chroma deploys (`SigmaPairHead` 384-DOF head wired, `BLEED_LOOP` §6 Gap 7), σ(c)≠c and the SAME fold verb gains a *complement-merge* — c⊕σ(c) → grey anchor m + chroma spread d — and the user steers d on an a/b disc while σ mirrors it for free. This must **never** be surfaced as a "find-the-partner / pick-the-complement" task: the decoder's internal `[c₀, σc₀, …]` adjacency must not be exposed as a pick, because that would pre-solve the user's work.

## 6. How the NN HELPS — the assistive functions

The NN is the user's *fidelity crew*; the user is the *only colorist*. Lead with the two landed pillars; mark the rest honestly.

- **GROUND** [LANDED] — *lives in 16².* The user is handed ONE coherent 256-cell grid, not 64 flickering grades. The NN sum-pools the 64 per-frame palettes (permutation-invariant collapse, `Collapse.hs`/`Bures.hs`, `LOOK_NN` Def 30/Thm 9) and places anchors at the Lloyd–Max ceiling behind a deterministic argmin index map (`LookNet`/`Significance.hs`), every cell real and populated. Fidelity is then **walled off and non-editable** — exactly one right answer the user can never get wrong — so all effort lands on the no-single-answer LOOK. *Scaffold-not-automate:* the index map is the single right answer (Thm 18); the user authors look, not fidelity. SEEING all 256 trustworthy cells at once is exactly what "trust the floor, author the look" needs.
- **REVEAL-as-RETENTION** [LANDED keystone] — *lives in 2⁸.* Every cell ships grey `(L,0,0)` but its chroma `(0,a,b)` is banked, not deleted (`Obfuscation.hs`, Thm 14, `L45.1` grayscale-truth, 332 tests green). The user authors TONE first on the grey loom (folds/splits) **knowing colour was never lost**. *Scaffold-not-automate:* d is banked by the user's own fold; the NN deletes nothing.
- **REVEAL-the-colour** [SPEC — dormant, gate on `Obfuscation`→`ColorBleed`] — *lives in 2⁸ + one bleed dial.* One scalar β progressively un-hides the banked chroma, **tone-forced-before-colour by V₊⊥V₋ orthogonality** (Thm 15/16/18, L52.8). The NN computes the reveal field; the **USER sets the amount**. `ColorBleed.hs`/`ChromaAllocation.hs` are contract-first, **not yet built** — today's depth-8 grey head has δ≡0 so β is inert (`BLEED_LOOP` §6 Gap 7). *Scaffold-not-automate:* β_user is the user's; the spec returns only `admissible(β)` + a dismissible β* (L52.10).
- **SUGGEST** [SPEC — not landed] — *lives in 4⁴ + 2⁸.* Dismissible, never-actuated scent: a posterization-depth hint on the quadtree, a waterfilling next-reveal proposal (largest hidden chroma χ_p first, L52.6) on the binary spine. Returned only as a predicate + dismissible suggestion, never committed. Capture loop telemetry **before** wiring the conditioning functional (per `feedback_dither_abstraction`; `BLEED_LOOP` §6 Gap 1). *Scaffold-not-automate:* by construction it self-actuates nothing.
- **DIVERSIFY** [LANDED groundwork] — *lives in the gallery.* A per-user gallery of *maximally-different* authored looks to keep/swipe, built on a diversity (DPP/QD) objective, **not** an MSE argmin (`Preference.greedyGallery`/`dppLogDet`, `Bures.buresDistanceSq`), so the loop inclines toward "make another" (Thm 17 engagement attractor, not a contraction onto a point). *Scaffold-not-automate:* the NN lays out alternatives; the user keeps or swipes.

In every case the NN computes scaffolding (fidelity floor, retention bank, reveal field, next-move scent, alternatives) and **actuates nothing** — every fold, pick, reveal-amount, and keep is the user's finger.

## 7. The user flow (SEE → NAVIGATE → AUTHOR → emit)

1. **Shoot.** The user captures a 64-frame colour burst. *NN:* COLLAPSE pools the 64 per-frame palettes into ONE global 256-colour palette and GROUNDS it (argmin index map + Lloyd–Max + significance). The user is handed a coherent floor, not 64 flickering grades.
2. **SEE (16²).** The Loom-Map opens: all 256 cells at once, sorted dark→light, every cell a trustworthy tonal anchor. The user reads the whole look in one glance and locates where the work is. *NN:* presents truth; may dismissibly highlight a busy region; re-sorts nothing.
3. **NAVIGATE (4⁴).** The user taps a crowded quadrant open to descend into the darks (or pulls the whole palette coarser for a flatter, graphic look). *NN:* keeps every coarsened super-cell real and populated; whispers E[d] ("~K levels"); guarantees committed folds compile losslessly to the binary spine.
4. **AUTHOR (2⁸).** Inside that region the user FOLDS pairs (press-hold-drag → m=½(c₀+c₁), d banked) and SPLITS to undo (lossless), settling at their own posterization N. *NN:* guarantees the fold/split round-trip, banks d (chroma retained), suggests the next coarse-to-fine fold via `goldenDecay` — dismissibly, never firing it.
5. **(Later, [SPEC]) REVEAL.** With tone authored on the grey skeleton, the user drives one bleed dial β; banked chroma reveals **in place** on the position-stable map, tone-forced-before-colour. *Dormant until the σ-pair head ships.*
6. **Emit.** The user emits the **GIF + its 256-colour palette** — a single deliberate grade across the whole burst. *NN:* renders the 64³ GIF from the authored palette; DIVERSIFY offers the next maximally-different look to keep or swipe, inclining the user to make another.

## 8. Form follows function (the principle, applied)

**Function leads.** The CORE function is "author ONE coherent look on a grounded, reveal-able 256-palette." Everything else is downstream of that single job-to-be-done: the user authors the LOOK; the NN owns the FIDELITY. The three forms of 256 are **not** chosen for elegance and then assigned meanings — each is the surface one capability of that job *needs*: SEEING needs simultaneity-with-meaningful-position → a 16×16 grid; NAVIGATING needs spatial coarse-to-fine → a quadtree; AUTHORING needs a single-gesture binary commit → a Haar pair-tree. The same 256 wears three faces because the one job has three hand-grips.

**What this DEFINES for the build** — which form each NN output surfaces in, and what the NN must expose to serve each function:

- **COLLAPSE → 16²:** expose the ONE global 256-palette as the deployed grid, 1:1, L-sorted dark→light. *(landed: `Collapse.hs`/`Bures.hs`.)*
- **GROUND → 16²:** expose each cell as a non-editable, populated, argmin-correct representative; expose **no** fidelity controls. *(landed: `LookNet`/`Significance.hs`, Thm 18.)*
- **RETAIN → 2⁸:** expose the fold/split as the lossless authoring verb (m, d banked); expose the fold counter (256→N), not the depth. *(landed: `PairTree`, `Obfuscation.hs` Thm 14.)*
- **NAVIGATE → 4⁴:** expose a 4-ary zoom lens over the committed binary folds; present Quad4 honestly as a chosen (possibly lossy, 513-DOF) view, never as a free lossless regroup. *(landed substrate `PairTree`; lens `Quad4.hs`/`Quad4Fit.hs`, fork unresolved.)*
- **REVEAL → 2⁸ + one bleed dial:** expose a single β that scales banked chroma in-place on a β-invariant grey skeleton; tone forced before colour. *(spec, dormant — wire `SigmaPairHead` 384-DOF head first; `ColorBleed`/`ChromaAllocation` not yet built.)*
- **SUGGEST → 4⁴ + 2⁸:** expose only a dismissible predicate + suggestion (depth hint, waterfilling next-reveal), never a committed action; capture telemetry before wiring the functional. *(spec.)*
- **DIVERSIFY → gallery:** expose a QD/DPP-diverse set of authored looks to keep/swipe, never an MSE argmin. *(landed groundwork: `Preference.hs`, `Bures.hs`.)*

The build constraint that holds across all of them: the NN **suggests, the user authors** — `S_θ` is a section, `F_U` is the only writer, and there is no auto-collapse button. Form follows function, and the function is a user with their finger on the fold.
