# V2.1 state and next steps (handoff notes, 2026-06-30)

This is the session handoff for SixFour V2.1, the pre-collapse probabilistic field. Read it first,
then the design docs listed at the bottom. Everything below is committed on `master`.

## What V2.1 is

Per 64x64 bin a colour channel is a PROBABILITY CURVE, stored energy-first (Q16 over 256 value
levels). The GIF89a the user sees is the COLLAPSE (the mode, argmin energy, `collapseQ16`). The model
trains on the curves, not the byte. V2.1 sits ABOVE the V2 latent's encode boundary, so it reopens no
locked V2 decision (`collapse . curve == the sRGB byte the V2 boundary already consumes`).

Mathematically: a fibre bundle. Base lattice B = (Z cap [0,63])^3 over (x,y,t); the fibre over each
base point is three Q16 energy curves of length 256; the whole field is a dense rank-5 tensor
`[T=64, Y=64, X=64, 3, 256]` (768 MB). The export is the time-pooled `[y, x, 3, 256]` slice (50 MB).

## What is built and verified (4 build steps)

1. SPEC (Tier 0). `SixFour.Spec.V21Field` (collapse, opponent delta, octant spine reusing OctreeCell,
   metric, S/K floor, read-depth, the empirical-histogram capture model) and `SixFour.Spec.V21FieldUI`
   (the UI cell-count layer: `budgetCells` Morton quadtree + exact-integer Hamilton `apportion`, and
   `allocateWidgets`, the OPPOSITION law where widgets take pairwise-distinct cell counts via a
   staircase repulsion, feasible iff total >= k(k-1)/2). `cabal test` = 1289, all laws QuickCheck'd.

2. SWIFT PORT + WIDGETS (Tier 2, behind `Feature.v21Capture`, now ON). `V21FieldUI.swift` is the
   byte-exact integer port of the count layer. `V21WidgetSurface.swift` is the two widgets: MODE
   (crisp collapse fills, zero bleed = the GIF) and UNCERTAINTY (heat splats whose bleed grows with
   spread, drawn translucent and oversized so neighbours OVERLAP). The bleed interacts; the counts
   oppose. One shared budget split distinctly by `allocateWidgets`.

3. CAPTURE TO TENSOR + AIRDROP. `V21CaptureField.swift` builds the probability field from a committed
   burst and writes it as a zero-dependency NumPy `.npy` (int32, shape `[y,x,3,level]`); `V21Export`
   bundles GIF + tensor. The review bench (`CapturedReviewPhaseField`) has FIELD (open the widgets)
   and AIRDROP (share GIF + tensor) buttons.

4. LIVE METAL DISPATCH. `v21AccumulateHistKernel` now runs during capture (additive pass on the
   per-frame command buffer), accumulating the true camera fine-grid histogram into a per-burst
   buffer, pooled over t into `Surface.v21Counts`. The review bench prefers this camera-box field and
   falls back to the index-cube proxy when it is absent. Plumbing: `MetalPipeline.v21HistPSO` /
   `submitAsync(v21Hist:)` / `makeV21HistBuffer` / `poolV21Counts`; threaded through
   `CaptureSession.BurstResult` -> `CaptureViewModel.v21Counts` -> `SurfaceView.commit` ->
   `Surface.v21Counts`. Kept out of `CaptureOutput` on purpose (its Hashable/Equatable would do a
   50 MB compare in SwiftUI diffing).

Verification at sunset: `cabal test` 1289; `zig build test -Drequire_fixtures=true` 75/75; full app
`xcodebuild` iOS-sim arm64 TEST BUILD SUCCEEDED; all V2.1 Swift suites 15/15 on a simulator
(V21FieldUIGoldenTests, V21CaptureFieldTests, V21MetalParityTests); `numpy.load` round-trips the
tensor. The Metal parity test closes the prior audit gap "no runtime Metal == Zig golden" for the
value-independent laws (per-voxel sum == scale^2, uniform box -> spike, total preserved).

## Open items / next steps (ranked)

1. `s4_v21_palette_delta` does NOT exist. It IS `axisWeight T = pd`, so it blocks BOTH the third
   (Delta / temporal) widget AND the time-stretch of the movement-follow function. Open Q3 is the
   256-colour-set distance metric (L1 vs EMD vs reuse of `s4_board_mass`). Until it lands with a
   golden, the temporal surface falls back to pd = 1 (t treated like x,y). This is the single blocker
   for the temporal widget.

2. On-device camera parity for the live field. The Metal kernel's accumulation/indexing/layout are
   runtime-verified in the simulator, but the camera -> texture color path (the shared
   `ycbcr10VideoRangeToLinearSRGB`, reused from the trusted box-average kernel) is verified by reuse,
   not by an end-to-end camera shot, because this build machine has no camera. Do a real capture and
   confirm the FIELD widgets and the AirDrop'd `.npy` look right. The path is additive, gated, and
   nil-safe, so the shipped capture is unaffected if anything is off.

3. The 768 MB per-burst histogram buffer (nLevels = 256). Allocated transiently behind the flag and
   freed after pooling, nil-safe if allocation fails. Confirm it does not cause memory pressure on the
   target device during a burst; if it does, options are a coarser live nLevels or an accumulate-mode
   kernel that pools on-GPU into a single frame.

4. The movement-follow + stability (anti-flicker) functions and the render/bleed METAL-GPU module from
   `V2.1-UIUX-FUNCTIONS.md` are designed but not yet built. The current widgets render bleed on the
   CPU (Canvas), which is fine until a device Metal compositor is wanted.

5. Trainer ingest: the `.npy` is directly `numpy.load`-able (shape `[y,x,3,256]` int32 counts; energy
   face = total - counts, mass face = counts / total, matching `Spec.V21Field`). Wire it into the
   trainer corpus when training resumes.
