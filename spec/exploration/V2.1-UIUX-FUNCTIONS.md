# SixFour V2.1 UI/UX FUNCTIONS, the probability field made interactive

2026-06-29. Exploration tier. This is the **function design** for the V2.1 surface, not a view
sketch. The brief is literal: **form follows function, and the function is PROBABILITY.** The field
is `[T, Y, X, 3, 256]` Q16 energy curves (`SixFour.Spec.V21Field`). The UI is therefore a set of
**functions over that field**: it inherits the field's grid, its statistics, and its collapse. A
widget is not a rectangle, it is `region -> budget -> layout -> raster`, a pipeline whose every
stage is grid-aligned and distribution-driven.

Companion specs: `V2.1-PROBABILISTIC-FIELD.md` (the field), `V2.1-ZIG-FUNCTION-MAP.md` (the
kernels), `SixFour.Spec.V21Field` (the gated core: `collapseQ16`, `opponentI`/`labDeltaAt`,
`axisWeight`, `countsToEnergy`, `accumulateHist`, `liftOctList`). Existing surface:
`SixFour/UI/Screens/V21/V21FieldView.swift` (gated behind `Feature.v21Capture = false`). Existing
budget idiom to align with: `NudgePaintView` `CellBudget` (16-cube, 9 channels, Morton order, phi6
diagonal).

The whole surface stays behind `Feature.v21Capture`. Pure stages are Tier-0 spec candidates with
laws; render stages are Tier-2 Swift/Metal verified against goldens.

---

## 0. The four principles, stated as constraints on the functions

1. **Form follows function; the function is probability.** Every UI quantity is a statistic of a
   per-bin distribution: the mode `collapseQ16`, the spread (entropy / energy variance), the
   opponent delta `labDeltaAt`. No decoration carries no statistic.
2. **The UI follows the GRID.** Every function is defined on the base lattice
   `B = (Z cap [0,63])^3` and refines along the Morton / quadtree order already used by `CellBudget`.
   Sub-bin placement is forbidden; cells are unions of base voxels.
3. **A widget has CELLS to BUDGET and must FOLLOW THE MOVEMENT.** A widget owns an integer cell
   budget `N`. The budget is allocated where the distribution is most uncertain, and re-allocated as
   the time cursor and the x,y deltas move, with a stability rule that prevents flicker.
4. **PIXEL BLEED follows the WIDGET.** Each widget has its own look: a per-widget render function
   maps a cell plus its local distribution to pixels, where the bleed radius is a function of the
   cell statistic (variance / entropy) and the colour is a function of the mode.

## 1. THE CELL-BUDGET FUNCTION

### 1.1 Objects

```
type Voxel  = (Int, Int, Int)                 -- (x,y,t) in the 64-cube (V21Field.Voxel)
data Region = Region { rx, ry, rt :: (Int,Int) }   -- half-open box [lo,hi) per axis, grid-aligned
type Cells  = Int                              -- a non-negative integer cell count (the budget)

-- A leaf of the quadtree/octree refinement of a Region: an aligned sub-box plus the budget it won.
data Plot = Plot { plotRegion :: Region        -- aligned sub-box, a union of base voxels
                 , plotMorton :: Int           -- Morton key of plotRegion's min corner (stable order)
                 , plotCells  :: Cells }        -- cells allocated to this plot
type Layout = [Plot]
```

`Region` is always grid-aligned: `lo` and `hi` are multiples of the current refinement stride, so a
`Plot` is a union of whole base voxels, never a fractional bin. This is principle 2 made type-level.

### 1.2 The statistic the budget chases

Cells go where the probability is **most uncertain**. The per-voxel uncertainty is the curve
spread; the per-region uncertainty is the mean spread over the region's voxels. Two equivalent
faces, pick by tier:

```
-- Energy-spread face (byte-exact, integer; Tier-0):
spreadE :: Curve -> Int                        -- range of energy = maximum - minimum  (flat curve -> 0)
spreadE c = if null c then 0 else maximum c - minimum c

-- Shannon-entropy face (real-valued diagnostic; off the byte path, Tier-2 display only):
spreadH :: [Int] -> Double                      -- normalised entropy of the count histogram in [0,1]
```

`spreadE` is the gateable one: it is a pure integer function of the stored energy curve, monotone in
"how undecided the bin is" (a confident spike has a large min-to-max energy range only if you read
the dual; use instead the **second-moment** form below for a true uncertainty, still integer).

```
-- Integer uncertainty weight of one voxel: total observation mass minus the mode mass,
-- i.e. how many sub-observations DISAGREED with the winner. 0 = unanimous (sharp), large = spread.
disagree :: Curve -> Int                        -- on an energy curve E = total - count, count = total - E
disagree c = sum [ ec - minimum c | ec <- c ]   -- sum of energy above the floor = sum of non-mode mass
```

`disagree` is the recommended budget weight: it is exactly `sum_l (total - count_l) - nLevels*min`,
an integer, zero on a unanimous (uniform fine cell -> spike, `lawHistUniformIsSpike`) bin, and it
grows with multimodality (the motion/occlusion case the field exists to keep). The region weight is
the sum over its voxels:

```
regionWeight :: (Voxel -> BinQ16) -> Region -> Int      -- sum of max-channel disagree over voxels
```

### 1.3 The function and its conservation law

The budget is a **deterministic largest-remainder (Hamilton) apportionment** of `N` cells across the
quadtree children of a region, weighted by `regionWeight`, with the Morton key as the deterministic
tie-break. This reuses the project's existing Hamilton width-share rounding idiom (the same
"round-to-integers conserving the total" pattern noted for the encoder width-sum).

```
budgetCells
  :: (Voxel -> BinQ16)     -- the field accessor (curve per voxel)
  -> Region                 -- the region to cover
  -> Cells                  -- N, the total cell budget
  -> Layout                 -- N cells distributed across aligned quadtree plots
```

Algorithm (pure, deterministic):

1. If `N <= 1` or the region is one base voxel, return a single `Plot` holding all `N` cells.
2. Split `Region` into its `<= 8` aligned octant children (quadtree in x,y when `rt` is a single
   frame; octree when the t-extent > 1). Children are produced in **Morton order** (the
   `mortonIndex` already in `NudgePaintModel`).
3. Compute `w_i = regionWeight child_i`. Ideal share `q_i = N * w_i / (sum w)`. Floor to `f_i`,
   distribute the `N - sum f_i` remaining cells one each to the children with the largest fractional
   remainders, ties broken by **smallest Morton key** (so the result is a pure function of the field,
   never of arrival order).
4. Recurse `budgetCells field child_i n_i` and concatenate.

**Conservation invariant (the law):**

```
lawBudgetConserves :: ... -> Bool
lawBudgetConserves field region n =
  n < 0 || sum (map plotCells (budgetCells field region n)) == n
```

Every cell is placed exactly once; none is created or dropped. This is the UI twin of
`lawHistTotalPreserved` (every fine sample counted once) and of the encoder width-sum.

**Grid-alignment law:**

```
lawBudgetGridAligned field region n =
  all (\p -> aligned (plotRegion p)) (budgetCells field region n)
```

every plot is a union of whole base voxels, Morton-keyed. This is principle 2.

**Determinism law:** `budgetCells` is a pure function, so `budgetCells f r n == budgetCells f r n`;
the Morton tie-break removes the only source of nondeterminism (equal-weight children).

Swift form (Tier-2, mirrors the spec; integer math so it matches the golden):

```swift
struct V21Plot { let region: V21Region; let morton: Int; let cells: Int }
enum V21Budget {
    static func allocate(field: V21FieldData, region: V21Region, cells n: Int) -> [V21Plot]
}
```

## 2. THE MOVEMENT-FOLLOW FUNCTION

### 2.1 What "movement" is in this field

Movement is not screen-pixel motion; it is **displacement along the field axes**, and the field
already prices it via `axisWeight`:

```
axisWeight :: PaletteDelta -> Axis -> Int          -- X=1, Y=1, T=pd   (V21Field, lawXyLinearTimeWeighted)
```

So a gesture is decomposed into x,y spatial steps (cost 1 each, isotropic) and t steps (cost = the
per-frame palette delta `pd`). A widget "follows the movement" by re-budgeting its cells toward the
region the cursor is moving **into**, with the t-axis cost stretched by `pd` so that a high
palette-churn frame transition pulls more cells than a still pan.

```
data Cursor = Cursor { cx, cy, ct :: Int          -- current voxel under the gesture
                     , vx, vy, vt :: Int }          -- per-axis velocity (signed voxel/step)

-- The movement-weighted region of interest: the box the cursor is heading into, sized by velocity,
-- with the t-extent scaled by the palette delta so churny transitions widen the temporal window.
focusRegion :: PaletteDelta -> Cursor -> Region
focusRegion pd (Cursor cx cy ct vx vy vt) =
  Region (span cx (vx * axisWeight pd X))
         (span cy (vy * axisWeight pd Y))
         (span ct (vt * axisWeight pd T))           -- t-window stretched by pd
  where span c v = (clamp (c - 0) , clamp (c + v + 1))   -- aligned + clamped to [0,64)
```

### 2.2 The re-budget function

```
track
  :: (Voxel -> BinQ16)   -- field
  -> PaletteDelta         -- pd for the current frame transition (the t weight)
  -> Cursor               -- gesture / time cursor
  -> Cells                -- N, the widget's budget
  -> Layout
track field pd cur n = budgetCells field (focusRegion pd cur) n
```

`track` is just `budgetCells` aimed at the movement-derived region: the same conservation and
alignment laws hold, so a moving widget can never leak or duplicate cells.

### 2.3 Stability (no flicker), the load-bearing part

A naive re-budget every frame flickers, because Hamilton remainders flip on tiny weight changes. The
fix is a **hysteresis lock** on the integer weights plus the existing Morton tie-break, both pure:

```
-- Smooth the per-child weight with an integer EMA (shift-based, byte-exact, no float):
emaWeight :: Int -> Int -> Int                     -- emaWeight prev cur = prev - (prev>>3) + (cur>>3)

-- A child only loses/gains a cell when its smoothed weight crosses the threshold by a HYSTERESIS
-- margin H; within +/-H it keeps last frame's allocation. (H in cells, e.g. 1.)
stableBudget :: Int -> Layout -> Layout -> Layout  -- H -> previous layout -> proposed -> committed
```

**Stability law (the anti-flicker contract):**

```
lawBudgetStableUnderSubThreshold field pd cur prev n =
  let next = stableBudget hMargin prev (track field pd cur n)
  in regionWeightChange prev next < hMargin  -->  cellSet next == cellSet prev
```

If the movement-induced weight change is below the hysteresis margin, the committed cell set is
**byte-identical** to the previous frame: no flicker. Above the margin, cells migrate, and because
the migration is Hamilton + Morton it is the minimal, deterministic reshuffle (a cell moves from the
lowest-priority losing plot to the highest-priority gaining plot, never a global churn). This is the
UI analogue of `lawReadDepthWellFounded`: a strictly-decreasing/strictly-increasing measure governs
when state changes, so the surface settles.

Swift form:

```swift
enum V21Track {
    static func layout(field: V21FieldData, paletteDelta pd: Int,
                       cursor: V21Cursor, cells n: Int,
                       previous: [V21Plot], hysteresis h: Int) -> [V21Plot]
}
```

## 3. THE WIDGET-LOOK / PIXEL-BLEED FUNCTION

### 3.1 The render type, shared by every widget

Each widget is a pure function from its plots plus the local distributions to a raster. The **bleed**
is the spread of a cell beyond its nominal pixel bounds, and it is **driven by the cell's
distribution**, so a confident cell renders crisp and an uncertain cell renders soft. Each widget
supplies its own `look` (the colour map) and its own `bleed` (the radius map); that is exactly
"pixel bleed follows the widget".

```
type Px        = SIMD3 UInt8                       -- an sRGB pixel
data Splat     = Splat { center :: (Double,Double) -- cell center in raster coords
                       , radius :: Double           -- bleed radius in pixels (>= half a cell = crisp)
                       , colour :: Px               -- the cell's look colour
                       , weight :: Double }          -- falloff peak

-- A widget look = (colour of a cell, bleed radius of a cell), both pure functions of the cell's
-- local distribution. The renderer is generic; only these two vary per widget.
data Look = Look
  { lookColour :: BinQ16 -> Px                      -- mode / opponent / entropy -> colour
  , lookBleed  :: BinQ16 -> Double }                 -- variance / entropy -> radius

renderWidget :: Look -> Layout -> (Voxel -> BinQ16) -> Raster
-- For each Plot: emit a Splat at the plot center with colour = lookColour(repCurve),
-- radius = lookBleed(repCurve); composite splats with a normalised falloff
-- (Gaussian-ish, but integer-LUT so it is byte-reproducible). Bleed > half-cell makes
-- neighbouring cells overlap (soft); bleed = half-cell is hard pixels (the GIF look).
```

The compositing falloff is an **integer LUT** (a fixed 256-entry radial weight table), not a
floating `exp`, so the raster is reproducible and golden-able. Bleed radius is clamped to
`[halfCell, maxBleed]`; at `halfCell` the splat is exactly one hard cell (no bleed), recovering the
crisp 64-cube look the existing `V21FieldView` ships.

### 3.2 Concrete widget looks

**(a) Mode widget, the sharp collapsed GIF.** Colour = the collapse byte; bleed = none.

```
modeLook :: Look
modeLook = Look
  { lookColour = \bin -> srgb ( collapseQ16 (binR bin)
                              , collapseQ16 (binG bin)
                              , collapseQ16 (binB bin) )   -- the GIF89a byte triple
  , lookBleed  = \_ -> halfCell }                           -- crisp, no spread
```

This is the proxy surface of section 4: every cell is its mode, hard-edged. It is the existing
`collapseImage` path, re-expressed as a degenerate (zero-bleed) widget.

**(b) Uncertainty widget, entropy bleed.** Colour = a grayscale (or heat) ramp of the spread; bleed
= the spread itself, so undecided cells visibly smear into their neighbours.

```
uncertaintyLook :: Look
uncertaintyLook = Look
  { lookColour = \bin -> grey (clamp01 (toUnit (maxChannelSpread bin)))   -- bright = uncertain
  , lookBleed  = \bin -> halfCell + bleedGain * toUnit (maxChannelSpread bin) }
```

`maxChannelSpread` uses `disagree` (1.2) or normalised entropy (display tier). A unanimous bin reads
dark and hard; a bimodal bin reads bright and bleeds, so the user **sees the doubt physically spread
across the grid**. This is the principled successor to the current flat `uncertaintyImage`.

**(c) Delta widget, opponent (L,a,b) motion bleed.** Colour = the opponent delta to the next frame
(`labDeltaAt` at the cell's mode level), mapped opponent-literal to a tint; bleed = the magnitude of
the delta, oriented along the dominant axis. Cells that are MOVING bleed; still cells stay put.

```
deltaLook :: Cursor -> Look
deltaLook cur = Look
  { lookColour = \bin -> let (dL, da, db) = labDeltaAt bin (nextFrameBin cur bin) (mode bin)
                         in opponentTint (dL, da, db)          -- a<->x, b<->y, L<->t (phi6)
  , lookBleed  = \bin -> halfCell + bleedGain * magnitude (labDeltaAt bin (nextFrameBin cur bin) (mode bin)) }
```

The delta widget is where `axisWeight`'s t-weighting surfaces visually: the bleed of a temporal
delta is scaled by `pd` (the palette delta), so a high-churn transition smears more. The phi6
pairing (a<->x, b<->y, L<->t) from `DualCube` chooses the bleed orientation, keeping the Delta
widget consistent with the nudge UI gauge.

### 3.3 The bleed-follows-distribution law

```
lawBleedMonotoneInSpread look bin1 bin2 =
  spread bin1 <= spread bin2  -->  lookBleed look bin1 <= lookBleed look bin2
lawModeWidgetIsCrisp = all (\bin -> lookBleed modeLook bin == halfCell)   -- the GIF look has zero bleed
lawBleedClamped look bin = halfCell <= lookBleed look bin && lookBleed look bin <= maxBleed
```

Bleed is monotone in uncertainty, clamped to the grid, and identically zero for the mode/proxy
widget. That is the formal statement of "each widget has its own look and the bleed follows the
widget".

Swift form (the renderer is Metal for the full 64-cube; the splat LUT is a Zig/Swift golden):

```swift
struct V21Look { let colour: (Int /*cell*/) -> SIMD3<UInt8>; let bleed: (Int) -> CGFloat }
enum V21Render {
    static func raster(field: V21FieldData, layout: [V21Plot], look: V21Look, edge: CGFloat) -> UIImage?
}
```

## 4. THE COLLAPSE-TO-GIF PROXY SURFACE

The user must be able to **see the GIF as a proxy for the model input**: the GIF is the mode slice
of the field, and the model trains on the whole field. The proxy surface is one function, pinned to
the kernel.

```
-- The proxy raster: every cell painted its collapse byte. This is renderWidget modeLook at full
-- budget (one cell per voxel), i.e. the zero-bleed degenerate of section 3.
proxy :: (Voxel -> BinQ16) -> Raster
proxy field = renderWidget modeLook (oneCellPerVoxel field) field
```

**The proxy contract (the law that ties the surface to `s4_v21_collapse`):**

```
lawProxyIsCollapse field voxel =
  pixelAt (proxy field) voxel
    == srgb ( collapseQ16 (binR (field voxel))
            , collapseQ16 (binG (field voxel))
            , collapseQ16 (binB (field voxel)) )
```

The proxy pixel at every voxel equals `collapseQ16` of that voxel's curves, channel by channel. On
device this is the existing `SixFourNative.collapseV21` path (`s4_v21_collapse`), so the surface is
**byte-exact against the kernel golden** (`v21_collapse_golden.json`). The UI states it plainly:

- **What ships (the proxy):** `collapse(field)`, the per-bin mode, hard pixels. This is the
  GIF89a; `collapse . curve == the sRGB byte V2Latent.encodeBoundary consumes`.
- **What the model sees (the full field):** the curves themselves (energy landscapes), surfaced by
  the Uncertainty and Delta widgets. The proxy is the mode (MAP estimate); it discards entropy and
  multimodality, which the other two widgets exist to show.

So the proxy surface is honest by construction: it renders the exact bytes the model collapses to,
while sitting beside widgets that render the information the collapse throws away. The toggle is the
existing `V21FieldMode` (SHIPS vs SPREAD), generalised to a third DELTA read.

## 5. BUILD PLAN

### 5.1 Pure functions (Tier-0 spec candidates, with laws)

Promote into a gated `Spec.V21FieldUI` (or extend `Spec.V21Field`), each with the named law,
QuickCheck'd in `Properties`:

| Function | Law(s) | Notes |
|---|---|---|
| `disagree` / `regionWeight` | weight `== 0` on a uniform spike (ties `lawHistUniformIsSpike`) | integer, byte-exact |
| `budgetCells` | `lawBudgetConserves`, `lawBudgetGridAligned`, determinism | Hamilton + Morton, reuses width-share idiom |
| `emaWeight` / `stableBudget` | `lawBudgetStableUnderSubThreshold` | integer EMA, no float; the anti-flicker proof |
| `focusRegion` / `track` | inherits budget laws; `axisWeight` t-stretch | `track = budgetCells . focusRegion` |
| `proxy` | `lawProxyIsCollapse` | ties UI to `s4_v21_collapse` golden |
| `lookBleed` family | `lawBleedMonotoneInSpread`, `lawModeWidgetIsCrisp`, `lawBleedClamped` | colour map stays display-tier; bleed radius is the gateable integer |

These reuse `collapseQ16`, `labDeltaAt`, `mortonIndex`, and the existing Hamilton rounding, so the
new gateable surface is small and rides proven cores.

### 5.2 Swift / Metal (Tier-2, verified against goldens)

- `V21Budget.allocate`, `V21Track.layout`: integer Swift mirrors of the pure functions, golden-gated
  against the Haskell (Haskell == Swift), like every other Tier-2 port.
- `V21Render.raster`: Metal compositor over the 64-cube, with the radial-falloff **integer LUT** as a
  Zig/Swift golden. The Mode widget (zero bleed) reuses the existing `PixelImage` path verbatim.
- All entry points stay behind `Feature.v21Capture` (currently `false`), mirroring `V21FieldView`.

### 5.3 What stays gated, and which audit gaps block which widget

`Feature.v21Capture = false`: the entire surface is statically unreachable in MVP1. Within that gate,
the audit findings block specific widgets:

1. **Missing `palette_delta` kernel (BLOCKS the Delta widget and the t-stretch of Movement-Follow).**
   `axisWeight`'s `T = pd` needs a real per-frame palette delta; the audit confirms `s4_palette_delta`
   is named in the function map but **not implemented**, and open Q3 leaves the 256-colour-set
   distance undefined. Until a gated `palette_delta` lands with a golden, the Delta widget must fall
   back to `pd = 1` (treat t like x,y), and `focusRegion`'s t-window cannot stretch. This is the
   single most important blocker for the moving/temporal surface.

2. **No runtime Metal == Zig golden (BLOCKS the full-field Metal compositor and the Uncertainty
   widget at scale).** `v21AccumulateHistKernel` in `Shaders.metal` is never called and has no
   parity certificate against `s4_v21_accumulate_hist`. Until a device golden pins Metal == Zig, the
   bleed compositor must run on the CPU path (the existing per-bin Swift entropy loop) or stay
   preview-only. The Metal `renderWidget` is gated on this golden.

3. **SIMT contract laws (`lawBinIsSimtFlat`, `lawOctantIsNeighbourhood`) are exploration-only.** They
   are not in the gated `Spec.V21Field`, so the budget/render code that assumes "every bin has exactly
   `nLevels`" rests on an ungated assumption. Promote them before shipping any widget that indexes
   curves by a fixed stride.

4. **Untested overflow refusal paths (`s4_v21_opponent_delta`, `s4_v21_counts_to_energy`).** The
   opponent/energy kernels refuse `RC_OUT_OF_RANGE` on i32 envelope breaches but those paths have no
   test. The Delta widget feeds `labDeltaAt` real device deltas; before it ships, add adversarial
   fixtures so a refusing kernel surfaces as an empty/flat cell, not a crash.

5. **End-to-end V2.1 path untested.** Camera -> Metal accumulate -> Zig energy -> Zig collapse -> UI
   has no integration test while the flag is off. The proxy surface (`lawProxyIsCollapse`) is the
   cheapest end-to-end anchor to build first: it exercises the whole pipeline and is already kernel-
   golden, so it is the recommended **first widget to wire** when `v21Capture` flips.

**Recommended order:** Mode/proxy widget (no new kernels, kernel-golden) -> Uncertainty widget (CPU
bleed, needs only `disagree` + budget laws) -> Delta widget (blocked on `palette_delta` +
opponent-refusal fixtures) -> Metal full-field compositor (blocked on the Metal == Zig golden).
