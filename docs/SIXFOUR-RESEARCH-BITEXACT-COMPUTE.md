# SixFour Research — BIT-EXACT TENSOR COMPUTE — deterministic/integer NN forward+back pass agreeing across Zig CPU + Metal GPU (+ Mac MLX trainer)

> Deep-research output (cited, adversarially verified) from the 2026-06-18 canonical-core workflow.
> Informs `docs/SIXFOUR-CANONICAL-PATH.md`. External claims are design inputs until grounded in-repo.

## Hard constraints on the canonical core (per this axis)
- INTEGER/Q16 IS THE ONLY BIT-EXACT CROSS-DEVICE CONTRACT. Bit-exact float across Zig-CPU and Metal-GPU is not achievable in general (FMA contraction + reduction reassociation + 2-ULP transcendentals). Therefore every value that must agree byte-for-byte across devices (collapse, board mass, σ-pair leaf reconstruction, argmax keys) MUST live in Q16 integer space; float is permitted only behind a 1e-6 region-accept gate (the PaletteValue head), never as a cross-device equality.
- ALL NONLINEARS MUST BE FIXED-POINT INTEGER POLYNOMIALS / LUTs / DYADIC SHIFTS WITH SPEC-PINNED CONSTANTS (I-BERT/I-ViT pattern). No libm exp/sqrt/sin/sigmoid on either Zig or Metal in any path that must agree. The one exact primitive available is integer Newton sqrt (<=4 iters for INT32) — use it for any norm/variance step. The same integer constants are emitted from the Haskell spec to both Zig and Metal.
- REDUCTIONS MUST BE ORDER-INDEPENDENT BY CONSTRUCTION, NOT BY LUCK: use a Kulisch-style WIDE INTEGER accumulator (i64/i128 holding full Q16xQ16 products, single final round-shift) so sequential Zig and tree-reduced Metal produce identical sums; the accumulator width must be proven non-overflowing in the Haskell spec (width >= 2*frac_bits + ceil(log2 N) + sign). This is the mechanism that extends bit-exactness from the FORWARD pass to the BACKWARD/gradient-accumulation pass.
- BATCH-INVARIANCE IS MANDATORY AND CHEAP HERE: the σ-pair decoder and value head must use a FIXED reduction tree / fixed tile shape independent of how many palettes, frames, or A/B pairs are processed together. Because SixFour tensors are tiny and fixed-shape, this is near-free; bake the exact reduction order into the spec and forbid any kernel whose result depends on dynamic batch/tiling.
- MLX IS A WEIGHT FACTORY, NOT A NUMERICAL ORACLE. MLX float is nondeterministic even run-to-run on the same Mac and is dtype-dependent; the train->deploy contract MUST be 'MLX float weights -> deterministic Q16 quantizer (the spec) -> Zig/Metal integer forward pass is the reference', never 'MLX float == device float'. The Tier-1/Tier-2 split in the stack is therefore numerically load-bearing, not just packaging.
- ROUNDING MODE IS PART OF THE CONTRACT, PER-OP. On-device inference and any LeafOverride/btUpdate application use round-to-nearest-even (deterministic). Stochastic rounding is permitted ONLY in the Mac trainer (where reproducibility is already not promised) OR via a counter-based PRNG keyed by (op-index, element-index) if it must be reproduced on-device. Spec must annotate each op's rounding mode.
- THE VERIFICATION HARNESS MUST BE TWO-MODE (TAO-style): exact byte-equality for integer kernels (Zig == Swift == Haskell == Metal), and operator-specific REGION acceptance = max(IEEE-754 portable bound, empirical p99.9) for the float head only. Add an adversarial hazard-trap golden that forces a*b+c vs fma(a,b,c) divergence and a parallel-vs-serial reduction divergence, asserting the build flags that pin them are actually in effect.

## Findings
### [high] For INDIVIDUAL IEEE-754 operations, the same inputs give the same result on CPU and GPU to a given precision; cross-platform float divergence is NOT inherent to IEEE-754 but comes from (a) FMA contraction (GPU fuses a*b+c, x86 CPU rounds twice), (b) reduction reassociation (parallel vs serial parenthesization), and (c) transcendental libraries (NVIDIA double sin guaranteed only to 2 ULP of correctly-rounded; no correct-rounding due to table-maker's dilemma). This is the authoritative primary source for the stack's 'float hex-IEEE754 within 1e-6' tier.
**Implication:** The stack's two-tier rule is correct and citable: keep Q16 integer as the bit-EXACT cross-device contract; for any float path (PaletteValue head, MLX-emitted weights), 1e-6 is achievable ONLY if you (1) force FMA contraction OFF or identical on both sides, (2) pin one reduction order, (3) avoid bare exp/sqrt/sin and use your own fixed-point polynomials (already the I-BERT pattern). Add a golden hazard-trap test that intentionally diverges a*b+c vs fma(a,b,c) to assert the contraction flag is pinned.
**Sources:** https://docs.nvidia.com/cuda/floating-point/index.html

### [high] The dominant cause of LLM/tensor nondeterminism is NOT concurrency/atomics but BATCH-SIZE-dependent reduction order ('batch invariance'). Making RMSNorm/matmul/attention batch-invariant (fixed tile sizes, fixed split-SIZE not split-count, per-element reductions) yields bit-identical outputs — Thinking Machines got all 1000/1000 completions identical on Qwen3-235B, at ~20% perf cost vs cuBLAS.
**Implication:** The σ-pair decoder forward pass (384-DOF → 256 leaves) must be written batch-invariantly: a fixed reduction tree over the 128 generators, independent of how many palettes/frames are processed at once. Since SixFour's tensors are TINY and fixed-shape (no dynamic batch), batch-invariance is essentially FREE here — bake the reduction order into the Haskell spec as the source of truth and assert Zig and Metal both walk that exact tree. This is a near-zero-cost win the stack should claim explicitly.
**Sources:** https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/

### [high] Integer-only transformers (I-BERT) run inference with NO floating point and NO dequantization, but the NONLINEARS are deterministic APPROXIMATIONS, not exact: i-exp (softmax) 2nd-order poly, max error 1.9e-3; i-GELU max error ~1.8e-2; integer LayerNorm uses an EXACT integer Newton sqrt that converges in <=4 iterations for any INT32. I-ViT's Shiftmax/ShiftGELU use bit-shift dyadic arithmetic for the same goal.
**Implication:** Any nonlinearity in the σ-pair decoder or value head (softmax over A/B logits, any GELU/sigmoid, any normalization) is bit-exact across Zig+Metal ONLY if implemented as a fixed-point integer polynomial / LUT / dyadic shift with the SAME constants on both sides — exactly the I-BERT/I-ViT recipe. Integer Newton sqrt (<=4 iters) is the canonical exact building block for any L2-norm/variance step. Pin coefficients in the Haskell spec; golden-vector each nonlinear. Do NOT call libm exp/sqrt on either side.
**Sources:** https://ar5iv.labs.arxiv.org/html/2101.01321, https://arxiv.org/pdf/2207.01405, https://arxiv.org/html/2501.13379

### [high] Exact, order-independent reproducible reductions ARE achievable via Kulisch long accumulators (a wide fixed-point register sized to hold the full dot-product range; associative and error-free except one final rounding). Historically considered impractical for binary64 (Kulisch's 4288-bit accumulator), but modern reproducible-summation work shows <38% overhead at scale, and for low-precision/fixed-point inputs the accumulator is small.
**Implication:** For SixFour's Q16 matmuls/dot-products, a Kulisch-style wide INTEGER accumulator (e.g. i64/i128 holding Q16*Q16 products with no intermediate rounding, single final shift-round) makes the reduction EXACTLY order-independent — so Zig CPU sequential and Metal GPU tree-reduced sums agree bit-for-bit regardless of thread layout. This is the rigorous way to get the back-pass (gradient accumulation) bit-exact, not just the forward pass. Recommend: spec the accumulator width = ceil(log2(N)) + 2*frac + sign headroom, prove no-overflow in Haskell.
**Sources:** https://people.eecs.berkeley.edu/~biancolin/papers/arith17.pdf, https://www-pequan.lip6.fr/~graillat/papers/SIAMEX14.pdf, https://dl.acm.org/doi/10.1007/978-3-319-31769-4_11

### [high] MLX float on Apple Silicon is NONDETERMINISTIC even across runs ON THE SAME DEVICE, and is dtype-dependent: a 1-row vs all-rows matmul differs by ~1449 on M2; float32 shows consistent small errors, float16/bfloat16 can hit ~1% or NaN over many ops. Quantized integer models (Q8_0 etc.) achieve perfect reproducibility. Root cause = Metal kernels' dynamic tiling / adaptive reduction order (batch-invariance failure), an implementation choice, not a math limit.
**Implication:** CRITICAL for the Tier-1 Mac MLX trainer: you CANNOT assume MLX-computed float weights/activations are reproducible, even Mac-to-Mac or run-to-run. So the train->deploy parity contract must NOT be 'MLX float == device float'. Instead: (1) train in float, but EXPORT to Q16 via a deterministic quantizer that IS the spec, and (2) make the on-device Zig+Metal integer forward pass the bit-exact reference — MLX is only a weight FACTORY, never a numerical oracle. This vindicates the stack's existing 'MLX trainer Mac-only / Zig core = source of truth' split; make it explicit that MLX outputs are quantized through a golden-gated path before they're trusted.
**Sources:** https://adityakarnam.com/mlx-non-determinism-apple-silicon/, https://arxiv.org/html/2601.19139v1

### [medium] Metal gives the precise levers needed: default math mode is 'fast' (allows reassociation, FP contract, no-NaN/Inf assumptions); fma() forces a single fused op; the metal::precise namespace gives tighter library functions; 'Allow Contract' controls cross-statement multiply-add fusion; and [[invariant]] guarantees a calculation stays bit-stable — but [[invariant]] is documented for vertex POSITION output, not general compute, so it cannot be relied on for kernel determinism.
**Implication:** For the Metal side of any FLOAT path: compile with fast-math/contraction DISABLED, use explicit fma() (or explicitly avoid it) to MATCH whatever Zig does, and use metal::precise for any transcendental. Do not rely on [[invariant]] for compute determinism. For the INTEGER path none of this applies — int ops in MSL are exact two's-complement, which is why Q16 is the safe contract. Action: add a build-flag assertion + a golden test that the Metal integer kernel byte-matches Zig on the reference vectors, and a separate float-tolerance test (1e-6) for any float head.
**Sources:** https://developer.apple.com/videos/play/wwdc2016/606/, https://developer.apple.com/forums/thread/131733, https://developer.apple.com/metal/Metal-Shading-Language-Specification.pdf

### [high] Stochastic rounding (used by NITI and most low-bit integer TRAINING to recover accuracy) is intentionally NON-deterministic and breaks bit-exact reproducibility unless the PRNG state is checkpointed; it is recommended for training only, with round-to-nearest for inference. Counter-based / LFSR PRNGs can make it reproducible if the seed+counter are part of the contract.
**Implication:** If the back-pass ever uses stochastic rounding for Q16 gradient accumulation, bit-exactness is LOST unless the RNG is a counter-based PRNG keyed by (op-index, element-index) so the same draw is reproducible on Zig and Metal. Recommendation: for the on-device path use round-to-nearest-even (deterministic) for any inference and any LeafOverride application; reserve stochastic rounding for the Mac trainer ONLY, where reproducibility is already not promised. Spec the rounding mode per-op.
**Sources:** https://arxiv.org/pdf/2009.13108, https://cloud.google.com/blog/topics/developers-practitioners/why-stochastic-rounding-is-essential-for-modern-generative-ai, https://arxiv.org/pdf/2511.00874

### [high] A whole research line now FORMALIZES the 'bit-exact OR bounded' dichotomy: TAO/optimistic-verification (arXiv 2510.16028) verifies NN outputs up to operator-specific acceptance regions by combining portable IEEE-754 worst-case bounds with empirical error-percentile thresholds, because vendor kernels legitimately reorder reductions and fuse operators. DiFR and Verde build inference verification despite nondeterminism.
**Implication:** This gives the stack a principled VERIFICATION methodology rather than just golden vectors: define per-operator acceptance regions = max(IEEE-754 portable bound, empirical p99.9 error) for each FLOAT op, and assert Zig vs Metal vs (optionally) MLX fall inside them; keep STRICT bit-equality only for the Q16 integer ops. Concretely: extend the existing golden-vector harness with a two-mode oracle — exact-equal for integer kernels, region-accept for float kernels — mirroring TAO. This is the citable justification for the stack's '1e-6-bounded' float tier.
**Sources:** https://arxiv.org/html/2510.16028v1, https://arxiv.org/html/2511.20621, https://arxiv.org/pdf/2502.19405

## Risks
- OVERCLAIMING BIT-EXACT FLOAT: it is tempting to advertise 'byte-exact NN across CPU+GPU' broadly, but primary sources (NVIDIA, MLX-on-Apple) show float bit-exactness across Zig-CPU and Metal-GPU is not generally attainable. Only the INTEGER path is truly byte-exact; the float head is 1e-6-bounded. Mis-stating this in docs/STATUS.md would be a verifiable false claim and would fail the doc-claims gate's spirit.
- BACK-PASS DETERMINISM IS HARDER THAN FORWARD: most cited determinism results (Thinking Machines, batch-invariant kernels) target INFERENCE. Gradient accumulation adds more/larger reductions and (in integer training) stochastic rounding. If a trainer ever runs partly on Metal, the back-pass can diverge even when the forward pass agrees; keep training float on MLX/Mac and quantize, rather than promising a bit-exact on-device backward pass.
- KULISCH ACCUMULATOR WIDTH / OVERFLOW: a wide integer accumulator only stays exact if it cannot overflow. An off-by-one in width reasoning (forgetting product doubling of frac bits, or log2(N) headroom) silently wraps in Metal (two's-complement) and breaks agreement. The width proof must be in the Haskell spec and golden-tested at the extreme-value corners, not assumed.
- SPEC-DRIFT IN NONLINEAR CONSTANTS: I-BERT-style polynomial/LUT coefficients must be byte-identical in Zig and Metal. If they are hand-duplicated rather than codegen'd from the single Haskell spec, they will drift and produce a silent ~1e-3 mismatch that passes loose tests but fails strict integer equality. Enforce single-source codegen for every constant.
- METAL FAST-MATH IS ON BY DEFAULT: if the float head is compiled without explicitly disabling fast-math/FP-contract, Metal will reassociate and contract, breaking even the 1e-6 bound against Zig. This is an easy-to-miss build-config footgun; it must be a pinned, asserted compile flag with a regression trap, analogous to the existing DEVELOPMENT_TEAM pinning lesson.
- [[invariant]] FALSE COMFORT: the [[invariant]] attribute is documented for vertex position, not general compute reductions, so relying on it to guarantee kernel determinism would be a subtle mistake. Determinism for compute must come from explicit fixed reduction order + integer arithmetic, not from an attribute.
