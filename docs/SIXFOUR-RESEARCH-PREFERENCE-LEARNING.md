# SixFour Research — ON-DEVICE PREFERENCE LEARNING: how several nets learn a user's taste from few A/B taps, on-device, privately

> Deep-research output (cited, adversarially verified) from the 2026-06-18 canonical-core workflow.
> Informs `docs/SIXFOUR-CANONICAL-PATH.md`. External claims are design inputs until grounded in-repo.

## Hard constraints on the canonical core (per this axis)
- The canonical core's A/B Bradley-Terry reward (btUpdate on 770-D θ) is identical to DPO; treat it as the baseline LOSS HEAD, not a fixed law. The core must expose θ/δ updates as a pluggable loss head so IPO (bounded squared loss) and KTO (unpaired binary) can be added without touching A/B capture, the σ-pair genome, or LeafOverride.
- All new estimators must operate in the existing 384-DOF generator-space δ / 770-D θ; no new palette representation is permitted. δ stays σ-locked and applied at INFERENCE (LeafOverride contract) — preference learning changes which δ, never the decode path.
- A device must hold a low-rank-Gaussian (Laplace) posterior over θ, not just a point estimate. Its mean drives the look; its covariance drives BOTH active pair selection and cold-start shrinkage. This posterior must be expressible in Q16 (mean + low-rank Cholesky factor) and golden-pinned Zig≡Swift≡Haskell.
- Pair selection becomes part of the core, not the UI: the next A/B pair is chosen by Double-Thompson / max-mutual-information from the posterior, decoded through the existing σ-pair path. The Q16 argmax key remains the tie-break. Random pairing is disallowed once the posterior exists.
- Anything leaving the device is ONLY a DP-noised, securely-aggregable summary of δ/θ (LoRA-shaped, 384-DOF), never raw taps, frames, or palettes. Local DP noise added on-device before any aggregation; the device receives back only the population aggregate, never another user's vector. This is a Tier-1 (Mac/server) aggregation consuming Tier-2 exports.
- Cross-device collaboration must be trust/similarity-weighted in δ-space (peers with nearby θ aggregate more strongly), not FedAvg, so an idiosyncratic user is not flattened to the mean.
- Cold-start initial θ = DP-aggregated population mean; blend to personal estimate by a closed-form empirical-Bayes shrinkage scalar that is a function of tap-count and the same posterior variance — one Q16 scalar, deterministic, golden-gated.
- Every learned quantity is PERCEPTUAL preference, never reconstruction error: the Lloyd-Max MSE ceiling does not bind here, and the JEPA-vs-statistical verdict (deterministic base + thin Bayesian residual) is preserved — the residual is a small Bayesian utility head (GP/Laplace), not a trained NN.

## Findings
### [high] DPO's loss is literally the Bradley-Terry MLE re-parameterized through the policy: -log σ(β log π_θ(y_w)/π_ref(y_w) - β log π_θ(y_l)/π_ref(y_l)). So the stack's existing A/B Bradley-Terry reward IS DPO's objective. Moving 'beyond Bradley-Terry' is not a rewrite; it is swapping the loss head while keeping the same A/B tap data and the same θ.
**Implication:** btUpdate on the 770-D θ is already a DPO-equivalent step. Keep it as the baseline reward head; the genuinely new options below are alternative LOSS HEADS over the same generator-space δ, not new data pipelines. No change to A/B capture, σ-pair genome, or LeafOverride is required to adopt them.
**Sources:** https://arxiv.org/html/2410.15595v2, https://parasdahal.com/notes/dpo-direct-preference-optimization/

### [high] IPO replaces DPO's log-sigmoid with a squared loss to a fixed margin: (Δ - 1/2)², where Δ is the same β log-ratio difference. This bounds the objective so KL regularization stays effective even when preferences are deterministic/near-deterministic — exactly the regime of a single user who almost always picks the same side. DPO overfits and pushes π_l→0 in that regime; IPO does not.
**Implication:** For a per-device θ updated from FEW taps with low-entropy taste, IPO's bounded squared loss is the safer on-device update rule than raw Bradley-Terry/DPO: it resists the degenerate overfit that collapses the LeafOverride δ when a user is consistent. Implementable in Q16 (it is a squared difference of two log-ratios) with no reference-model retraining — the δ is applied at inference, so π_ref is the frozen σ-pair decode.
**Sources:** https://arxiv.org/html/2410.15595v2, https://hippocampus-garden.com/preference_optimization/, https://arxiv.org/pdf/2409.11564

### [high] KTO (Kahneman-Tversky Optimization) needs only UNPAIRED binary labels (this tap was desirable / undesirable), not pairs. Loss uses a prospect-theoretic value with separate loss-aversion weights λ_w, λ_l around a reference offset z_ref. It matches or beats DPO from 1B–30B params and stays robust with up to 90% imbalance, implicitly ignoring hard/mislabeled examples. NOTE: contrary to some secondary blogs, the primary paper confirms KTO DOES use a frozen reference model (the z_ref offset is a log-ratio to π_ref).
**Implication:** KTO unlocks signal the A/B-only path throws away: a swipe-to-LOOK kept/exported = desirable; a look dismissed without compare = undesirable. These single-sided taps are far more frequent than explicit A/B pairs. Add a KTO head over the same θ/δ so the model learns from solo interactions, multiplying effective feedback per session. The σ-pair decode is the frozen reference; loss-aversion asymmetry (λ) is a 2-scalar add to the device config.
**Sources:** https://arxiv.org/html/2402.01306v3, https://arxiv.org/pdf/2402.01306, https://argilla.io/blog/mantisnlp-rlhf-part-7/

### [high] Choosing WHICH pair to show is itself optimizable: maximize expected information gain / mutual information between the user's response and the unknown taste location. Optimal querying is NP-hard, but mutual-information acquisition beats volume-removal, and Double Thompson Sampling for dueling bandits picks the comparison pair by sampling twice from the posterior over the preference matrix (O(K log T) regret), needing only a Laplace/Gaussian posterior over weights.
**Implication:** Do NOT show random A/B pairs. Maintain a cheap Gaussian (Laplace) posterior over the 770-D θ and select the next pair by Double-Thompson sampling two candidate looks from the generator space — this maximizes taps-per-bit-of-taste, the dominant constraint when feedback is sparse. This is a small deterministic kernel (sample θ twice from a diagonal/low-rank Gaussian, decode two looks via the existing σ-pair path, present them). Fits the Zig Q16 core: posterior = mean + Cholesky of a low-rank covariance; argmax stays the existing Q16 argmax key.
**Sources:** https://arxiv.org/pdf/2402.15757, https://arxiv.org/abs/1604.07101, http://papers.neurips.cc/paper/6157-double-thompson-sampling-for-dueling-bandits.pdf, https://arxiv.org/html/2605.26072v1

### [high] Gaussian-Process preference learning (Chu & Ghahramani, ICML 2005) models a latent utility from pairwise comparisons with Gaussian-noise-contaminated observations and is well-suited to SMALL data because it is non-parametric and yields a posterior with calibrated uncertainty; expected informativeness of a new pair = change in posterior entropy.
**Implication:** For the FEW-tap regime, a GP/Laplace utility over a low-dim projection of the 770-D θ gives both the taste estimate AND the uncertainty needed for active pair selection in one object — no NN training loop on device. This is the principled 'thin learned residual' the repo's JEPA-vs-statistical verdict already endorses: deterministic σ-pair base + a small Bayesian utility head, not a trained network. Lloyd-Max-style MSE concerns do not apply because the GP scores PERCEPTUAL preference, not reconstruction.
**Sources:** https://icml.cc/Conferences/2005/proceedings/papers/018_Preference_ChuGhahramani.pdf, https://journals.sagepub.com/doi/10.1177/02783649231208729, https://www.sciencedirect.com/science/article/abs/pii/S0004370225001195

### [high] Cross-user taste is aggregated privately by Apple-style Private Federated Learning: data stays on device, only model/LoRA-style updates leave, local DP noise is added on-device and then AMPLIFIED by secure aggregation (secure multiparty computation / anonymous aggregation), giving formal (ε,δ)-DP at population scale. Deployed by Apple to hundreds of millions of devices for media preferences, emoji, etc.
**Implication:** The cross-user 'taste prior' must be assembled by uploading ONLY a DP-noised, securely-aggregated summary of the 770-D θ (or its low-rank δ), never raw A/B taps or images. Local DP noise on θ + secure aggregation gives a population-mean taste vector with provable privacy. This is Tier-1 (Mac/server aggregation) consuming Tier-2 device exports; the device never receives another user's θ, only the aggregate prior. Keep it byte-budgeted: ship the δ (384-DOF generator space), not full palettes.
**Sources:** https://machinelearning.apple.com/research/learning-with-privacy-at-scale, https://research.google/blog/federated-learning-with-formal-differential-privacy-guarantees/, https://arxiv.org/pdf/2412.07962, https://www.intechopen.com/chapters/1190994

### [medium] Several devices can collaborate WITHOUT a central model by trust-weighted, similarity-based selection: 'Personalized Collaborative Fine-Tuning' uses model-similarity / prediction-similarity / validation-performance schemes to weight which peers' LoRA updates to aggregate, exchanging only low-rank adapters, and beats both FedAvg and purely-local fine-tuning when local data is diverse and scarce. DP-FedLoRA adds formal DP to the exchanged adapters.
**Implication:** 'Several nets learning one user's taste' maps to several DEVICES/sessions each holding a low-rank δ adapter, aggregated by SIMILARITY weight (users with nearby θ pull together, divergent users do not). This is the right answer to 'don't let one global model flatten an idiosyncratic user': trust-weighted aggregation in generator-δ space, not FedAvg. The σ-locked 384-DOF δ is already a LoRA-shaped low-rank object, so the adapter-exchange machinery applies directly with no new representation.
**Sources:** https://openreview.net/forum?id=bwo3GVsgOv, https://arxiv.org/html/2509.09097v1, https://arxiv.org/pdf/2405.04840

### [high] Cold-start is solved by empirical-Bayes / James-Stein shrinkage: estimate a population prior from everyone, then for a new user with little data the posterior is pulled toward that prior and 'unshrinks' toward the user's own estimate as taps accumulate. Shrinkage keeps sparse-data estimates grounded; James-Stein is the canonical special case.
**Implication:** Initialize each device's θ at the DP-aggregated population mean (the federated prior) and blend toward the personal estimate by an empirical-Bayes shrinkage factor that is a closed-form function of tap-count and posterior variance — exactly the GP/Laplace variance you already compute for active selection. This is one Q16 scalar (shrinkage weight) per update, fully deterministic and golden-gateable. No magic: the same posterior covariance drives BOTH pair selection and cold-start blend.
**Sources:** https://www.amazon.science/publications/addressing-cold-start-in-product-search-via-empirical-bayes, https://kiwidamien.github.io/shrinkage-and-empirical-bayes-to-improve-inference.html, https://www2.stat.duke.edu/~pdh10/Teaching/732/Notes/shrinkage.pdf

## Risks
- KTO/IPO/DPO all assume a frozen reference policy π_ref in their log-ratios. On this stack the natural reference is the deterministic σ-pair decode; if that reference is mis-specified the β log-ratio is meaningless. Verify the reference choice before adopting any of the three (note: secondary blogs wrongly claim KTO is reference-free — the primary paper says otherwise).
- Double-Thompson / mutual-information pair selection assumes a usable posterior over θ. With <~5 taps the posterior is dominated by the prior, so early selection is effectively prior-driven; do not over-engineer active selection before the cold-start shrinkage path is solid.
- DP noise scale vs. utility is a real tension: enough local DP noise to be private can swamp a 770-D θ summary from a small device population. Budget ε carefully and aggregate the low-DOF δ (384), not full θ, to keep signal-to-noise viable; validate at realistic device counts before promising a useful population prior.
- Trust/similarity-weighted federation can fragment into echo-chamber clusters (idiosyncratic users never pull toward anyone), undermining the cold-start prior. Need a floor weight to the global mean so collaboration degrades gracefully to FedAvg-like behavior when no similar peers exist.
- 'Personalized collaborative fine-tuning' (OpenReview bwo3GVsgOv) and DP-FedLoRA are recent and LLM/large-model framed; their empirical claims are NOT validated on a 384-DOF palette generator or on phone-class device counts. Treat the similarity-weighting SCHEME as the transferable idea, the benchmark numbers as out-of-domain (medium confidence).
- Most preference-optimization literature is LLM-centric; there is little direct evidence of DPO/IPO/KTO applied to tiny non-LLM nets or directly to a reward/utility head. The loss MATH transfers cleanly (they are scalar functions of two log-ratios), but adopt them as loss heads on the existing scalar utility, not as a port of an LLM training recipe.
- Quantization risk: IPO squared loss and KTO prospect value involve products and sigmoids that must stay bit-exact across Zig/Swift/Haskell. The squared and asymmetric-weight terms are new Q16 reassociation hazards (simd_sum order, @divFloor); they need fresh golden vectors before they can be trusted as canon.
